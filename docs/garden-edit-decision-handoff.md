# Garden Edit Decision Handoff

This handoff captures the dogfooding finding around `garden edit` and direct edits to `AGENTS.md`.

## Decision

Do not add `garden edit` right now.

Garden should keep context cards as normal Markdown files. Humans and agents can edit `.garden/context/*.md` directly, then run:

```sh
go run ./cmd/garden agents sync --apply
go run ./cmd/garden lint
```

The dogfooding flow proved this is enough for the current product loop.

## Evidence From Dogfooding

We created `.garden/context/agents-router.md` with the Garden CLI, then edited its Markdown body directly.

The card frontmatter became the source of truth:

```yaml
scope:
  - AGENTS.md
  - internal/agents/**
  - docs/compact-index-syntax.md
```

After running `garden agents sync --apply`, `AGENTS.md` correctly gained these routes:

```txt
|AGENTS.md:.garden/context/agents-router.md
|docs/compact-index-syntax.md:.garden/context/agents-router.md
|internal/agents/**:.garden/context/agents-router.md
```

`garden lint` passed afterward.

This supports the product bet: edit Markdown directly, sync the generated router, lint for correctness.

## AGENTS.md Generated Section Concern

There is no reliable way to physically prevent an agent from editing the Garden-managed section in `AGENTS.md`.

The practical guardrails are:

- Marker comments clearly identify the managed section.
- `.garden/context/agents-router.md` tells agents not to hand-edit the managed section.
- `garden agents sync --apply` rewrites the managed section from card frontmatter.
- `garden lint` detects stale or malformed Garden index state.
- Maintainers review diffs before commit.

This is enough for now. Do not add complex protection mechanisms unless dogfooding shows repeated real damage.

## What Not To Build Yet

- Do not add `garden edit` as a wrapper around normal Markdown editing.
- Do not add a lock file or hidden state to prevent edits.
- Do not add a second source of truth for card metadata.
- Do not make agents run an extra discovery command when `AGENTS.md` already contains the route map.

## Implemented Follow-Up

Garden now emits one explicit generated warning line inside the managed block:

```md
<!-- This section is generated by Garden. Edit .garden/context/*.md, then run garden agents sync --apply. -->
```

This keeps the guardrail visible in `AGENTS.md` without adding a new command, lock file, hidden state, or second source of truth.

## Next Steps

1. Keep dogfooding direct card edits plus sync/lint.
2. Watch whether the generated warning reduces accidental edits to the managed `AGENTS.md` section.
3. Do not add `garden edit` unless future dogfooding exposes a concrete gap in direct Markdown edits plus sync/lint.
